3DCompass Role and Openness
Technical Article

Abstract

3DCompass allows a person granted as platform administrator to create and manage 3DEXPERIENCE applications. This article is an overview of its capabilities, and what sort of openness it proposes

Role

3DCompass is the service in charge of managing the applications available from the Compass of a 3DEXPERIENCE platform. Some keys information about such an application:

  • It can be a native app ( require a local installation of the application), a web app ( open a new tab in the browser), or dashboard app ( launched in a dashboard) [1].
  • It must be associated with a Compass quadrant, North (Social and Collaborative apps), East (Information Intelligence apps), South (Content and Simulation apps) or West (3D Modeling apps). Only the apps of the selected quadrant are available to be launched.
  • It can be (or not) associated with a role to sort them. Only the apps of the selected quadrant, and the selected roles, are available to be launched. An application associated with no role is visible when no role is selected.
  • It can be available for all members granted to the platform instance, or for some of them.

The reader will find information about the Compass in the User Assistance documentation. The node is "3DEXPERIENCE Basics | Compass".

In the Compass there are two kinds of applications: the DS ones, and the third party apps also, named Additional App. When the formers come with DS products, the latters are those of the customers. Only people granted as platform administrator can create and manage third party apps.

The reader will find information about third party apps in the User Assistance documentation. The node is "Installation and Administration | 3DEXPERIENCE Platform | 3DEXPERIENCE Platform - Administration | Platform Management".

Platform Isolation

The 3DCompass service is not based on the multi-tenancy [2] principle. The 3DCompass service is the same whatever the platform instances of one 3DEXPERIENCE platform.

Secured Access

Like expressed by this whitepaper on our www.3ds.com web site [3], the security is on the heart of the 3DEXPERIENCE platform. When using 3DCompass service you have to be familiar with some points of security:

  • Authentication

    3DCompass, like all the services of the 3DEXPERIENCE platform, requires an authentication. The protocol is based on the CAS protocol, with a centralized management through the 3DPassport [4]. So, before using 3DCompass you must perform a login.

  • Cross-domain

    All the browsers recommended by the 3DEXPERIENCE platform prevent cross-domain access. The 3DEXPERIENCE servers solve this security point by implementing CORS.

    CORS (Cross-Origin Resource Sharing) enables secured, and controlled, cross-domain communications from the browser. For example it enables direct access to 3DCompass web services inside a widget from 3DDashboard. Let's see the referenced article [5] dealing with HTTP requests from a widget application.

  • Security Context

    Security context is a 3DSpace concept, so you do not have to take it into account when invoking 3DCompass services. But if the HTTP request contains a security context as header field, its value must be valid for the connected person.

REST Web Services

3DCompass only exposes web services to manage third party apps:

  • Creation,
  • Retrieving one or a collection
  • Edition,
  • Deletion,

Before using these web services, you are invited to read "About 3DCompass Web Services" [6]. The article is gathering the common rules for 3DCompass web services.

6WTags

There is no 6WTags associated with the 3DCompass resources.
