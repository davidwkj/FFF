What You Will Learn With This Use Case

This use case is intended to show how to create an add-in of a data model. An add-in is an object that enables to extend the data model by adding commands that will be available for all apps linked to the data model . Command headers are used to make the link between the add-in and the commands.

The CAAAfrGeoClippingAdnD Use Case

CAAAfrGeoClippingAdnD is a use case of the CAAApplicationFrame.edu framework that illustrates the AfrFoundation framework capabilities.

What Does CAAAfrGeoClippingAdnD Do

The CAAAfrGeoClippingAdnD use case creates an add-in of the CAAGeometry model. Is is made of a set of two "Clipping" commands, the Clipping by Box and the Clipping by Sphere commands, that are added in the Tools section of the action bar.

How to Launch CAAAfrGeoClippingAdnD

See the section entitled "How to Launch the CAAGeometry Use Case" in [1] for a detailed description of how this use case should be launched.

Do not type the module name on the command line, but type 3DEXPERIENCE instead. When the application is ready, do the following:

  • Click on 3D, the west quadrant of the Compass.
  • Select the app CAA:Create Geom with D-Afr.
  • Activate the Tools section of the action bar and check that the clipping commands are available.

Where to Find the CAAAfrGeoClippingAdnD Code

The CAAAfrGeoClippingAdnD use case is made of a declarative file located in the CAAAfrGeoWksAddinD.m module in the CAAApplicationFrame.edu framework:

InstallRootFolder\CAADoc\CAAApplicationFrame.edu\CAAAfrGeoWksAddinD.m\

where InstallRootFolder [4] is the folder where the API CD-ROM is installed.

The declarative file is an XML type file called CAAAfrGeoClippingAdnD.afr

Step-by-Step

To create an addin, there are five steps:

Creating the Module

Updating the IdentityCard

The IdentityCard of your framework that will contain the addin declaration needs to contain the following line: 

...
<toolPrerequisite name="MkDAfrTool"/>
...

This line allows MkCopyPreq to copy the needed prerequisites to process the declarative file of your app by mkmk.

Creating a Dedicated Module

The declarative file must be defined inside a dedicated module in order to be correctly processed by mkmk.

  • Creating the Imakefile.mk

    The Imakefile.mk file must contain the two following lines: 

    BUILT_OBJECT_TYPE=RUNTIME DATA 

SCRAMBLING_KEY=2

    These lines will activate an encryption of the declarative file by mkmk before putting it inside the RuntimeView.

  • Creating the Necessary Folder Hierarchy

    The declarative file must be defined inside a particular folder hierarchy that should be strictly respected because it contains all the necessary information to create the addin. 

    CAAApplicationFrame.edu
|
|
-------- CAAAfrGeoWksAddinD.m
              |
              |
              -------------- Imakefile.mk
              |
              |
              -------------- src
                              |
                              |
                              ------ resources
                                         |
                                         |
                                         -----ApplicationFrame
    • CAAApplicationFrame.edu is the name of the framework containing the declarative file.
    • CAAAfrGeoWksAddinD.m is the name of the module containing the declarative file.
    • Under the src folder, you must have a folder called resources and under its folder another folder called ApplicationFrame (this naming and hierarchy are mandatory).

    As we are creating an addin of a model, the folder hierarchy needs to contain the information concerning the model to extend:

    ApplicationFrame
  |
  |
  ----------- AfrWorkshop
                   |
                   |
                   ------- CAAAfrGeometryWks
                                 |
                                 |
                                 ----- AfrAddin
    • ApplicationFrame is the folder we previously created inside the src folder of the module.
    • AfrWorkshop means that we are adding a declarative file for a data model, also called workshop.
    • CAAAfrGeometryWks is the Id of the CAA geometry model that we are extending. This folder must have the exact name of the workshop Id.To obtain the exact name of data models, you can use a tool called Application Frame Structure Exposition command [2].
    • AfrAddin means that we are creating an addin.

Creating the Add-in Declarative file

The declarative file describes your addin. It must have the .afr extension in order to be correctly processed by mkmk. For this use case, the file is called CAAAfrGeoClippingAdnD.afr

Following is the content of the CAAAfrGeoClippingAdnD.afr file:

<?xml version="1.0" encoding="utf-8"?>

<Styles xmlns:syp = "http://www.3ds.com/xsd/SYP">

  <Template syp:name = "CAAAfrGeoClippingAdnDHeaders" Target = "CATCmdHeadersList">
    <CATCmdHeadersList>
      <CATCommandHeader ID = "CAAAfrClippingByBoxHdrD" ClassName = "CAADegClippingByBoxCmd" SharedLibraryName = "CAADegGeoCommands" ResourceFile = "CAAAfrGeoClippingAdnHeaderD" Available = "1"/>
      <CATCommandHeader ID = "CAAAfrClippingBySphereHdrD" ClassName = "CAADegClippingBySphereCmd" SharedLibraryName = "CAADegGeoCommands" ResourceFile = "CAAAfrGeoClippingAdnHeaderD" Available = "1"/>
    </CATCmdHeadersList>
  </Template>

  <Template syp:name = "CAAAfrGeoClippingAdnDAccess" Target = "CATCmdAddin">
    <CATCmdAddin>
      <CATCmdContainer Name = "AfrGeneralSection">
        <CATCmdContainer Name = "CAAAfrClippingTlbD">
          <CATCmdStarter Name = "CAAAfrTClippingBoxStr" Command = "CAAAfrClippingByBoxHdrD"/>
          <CATCmdStarter Name = "CAAAfrTClippingSphereStr" Command = "CAAAfrClippingBySphereHdrD"/>
        </CATCmdContainer>
      </CATCmdContainer>
    </CATCmdAddin>
  </Template>

</Styles>

The first line adds a basic XML tag that indicates the file is a XML one. Next, you have to add the root node of the XML file: the Styles node. This node indicates to the parser that it must analyze this file as a declarative file.

As you can see, the content of the xml file is made of two sections:

  • CATCmdHeadersList section : enables to define your command headers.
  • CATCmdAddin section : enables to define the layout of your commands.

These two sections will be fully described in the following chapters.

Creating the Command Headers

Following is the XML sample corresponding to this section:

<Template syp:name = "CAAAfrGeoClippingAdnDHeaders" Target = "CATCmdHeadersList">
  <CATCmdHeadersList>
    <CATCommandHeader ID = "CAAAfrClippingByBoxHdrD" ClassName = "CAADegClippingByBoxCmd" SharedLibraryName = "CAADegGeoCommands" ResourceFile = "CAAAfrGeoClippingAdnHeaderD" Available = "1"/>
    <CATCommandHeader ID = "CAAAfrClippingBySphereHdrD" ClassName = "CAADegClippingBySphereCmd" SharedLibraryName = "CAADegGeoCommands" ResourceFile = "CAAAfrGeoClippingAdnHeaderD" Available = "1"/>
  </CATCmdHeadersList>
</Template>

The section that describes the command headers is referenced by a name valued with the syp:name attribute. This name must follow a naming convention: xxxHeaders where xxx is the name of your addin (the same as the name of the XML file). The Target attribute must be valued to CATCmdHeadersList as we are currently declaring command headers.

Then, the CATCmdHeadersList tag can have as many child tags as you want to declare command headers for your addin. The tag to declare command headers is the CATCommandHeader tag. It enables you to declare a command header by giving the following information:

  • ID: this ID is used to reference the command header (for example to references a particular command header when defining the layout of commands in the frame).
  • ClassName: this attribute is the class to instantiate to launch the command (this class should use the CATCreateClass macro).
  • SharedLibraryName: this attribute corresponds to the name of the shared library that contains the code of the command.
  • ResourceFile: this attribute corresponds to the name of the resource file associated with the command.
  • Available: this attribute indicates whether the command is available or not for the Normal mode (1 means available, 0 means hidden). Prefer using the attribute AvailabilityMode that allows you to define the command header availability for each mode (in fact Available="1" is similar to AvailabilityMode="Normal=YES").
  • AvailabilityMode: specify the availability of the command header for each mode. This attribute takes as parameter a string with the following syntax "Mode1=YES Mode2=NO Mode3=...". Note that if you don't specify an availability for a particular mode, this mode will use its default availability. The different modes for which the availability can be customized are the following:
    • Normal
    • Editability
    • SmallScale
    • LargeScale
    • Offline

Arranging the Commands in a Section of the Action Bar

The aim is to organize the commands in the frame. In this example, we will add our commands in the Tools section of the action bar.

We can see that the Clipping commands of our addin have been added in the Tools section.

It is mandatory to organize your commands in containers to be able to insert them in a section of the action bar.
The hierachy of containers is illustrated in the picture below:

Following is the XML sample corresponding to this section:

<Template syp:name = "CAAAfrGeoClippingAdnDAccess" Target = "CATCmdAddin">
  <CATCmdAddin>
    <CATCmdContainer Name = "AfrGeneralSection">
      <CATCmdContainer Name = "CAAAfrClippingTlbD">
        <CATCmdStarter Name = "CAAAfrTClippingBoxStr" Command = "CAAAfrClippingByBoxHdrD"/>
        <CATCmdStarter Name = "CAAAfrTClippingSphereStr" Command = "CAAAfrClippingBySphereHdrD"/>
      </CATCmdContainer>
    </CATCmdContainer>
  </CATCmdAddin>
</Template>

The syp:name attribute for this section must follow a naming convention: xxxAccess where xxx is the name of your app (the same as the name of the XML file). The Target attribute must be valued to CATCmdAddin.

The Template must have for unique child a CATCmdAddin tag.

Then, the CATCmdAddin tag can have as many child CATCmdContainer tags to define the hierarchy of containers. In our example, we have a level of container for Tools section of the action bar, that is referenced by the pre-defined keyword AfrGeneralSection. Then, in the Tools section we define a group of commands, that will contain our two commands. The CATCmdContainer tag just requires a name attribute that enables to link the container with its corresponding resource file. 

In the last level of container, the CATCmdStarter tag enables to declare the commmands for the container. This tag requires two attributes, a name attribute and the command attribute that contains the Id of the command header.

Providing the Add-in Resources

You should provide the resources for the commands of your addin : the command header resources in the command header resource files: titles, messages, icons. You can refer to the article "Creating an App with a Declarative File"[3] in which the declaration of resources is described in detail. The resources files for this use case are: CAAAfrGeoClippingAdnHeaderD.CATNls and CAAAfrGeoClippingAdnHeaderD.CATRsc.

In Short

An add-in to a data model enables to add commands without modifying any code. You simply need to create an XML type declarative file with the .afr extension that enables to define your addin commands. The declarative file should be located in a specific path of folders that contains information on the data model to extend.
