Revisit existing Tools/Options functionality

When the command "Preferences" is launched, a dialog box is displayed containing a list of apps and their associated properties, organized in tabs.

The proposed options in this dialog panel,  allows  system administrators or end-users to customize apps and then to control the runtime behavior of  their 3DEXPERIENCE session

 

A New preference panel has been developed and is customisable for CAA Apps.

This new command gathers in one single dialog box all the app properties that can be customized and is more easily customisable thanks to xml files, without need to develop any code..

Retrieve Workbench / App node

The workbench is retrieved by using implementation of the API CATIWorkbench for a particular solution.

Create Tab Pages

The interface CATIUserSettings is dedicated to create a User – Preferences property page.

Each property page of the User – Preferences command corresponds to an implementation of this interface.

The Interface CATIGenericFactory  enables to generically define factories.  The CAA Application has to provide a factory to enable the instantiation of a component by User – Preferences

STEPS:

  • Create the factory interface which derives from CATIGenericFactory.
  • Implement this factory interface using macros.
  • In the interface dictionary, declare that CATUserSettingsManager implements factory interface.
  • Implement the CATIUserSettings interface.
  • In the factory dictionary, declare that the factory creates an instance of the CATIUserSettings implementation.
  • Define location of Property Page in .CATRsc file using Solution & Workbench / App name.

 

Setting Attributes and Repositories

Setting attributes provide an easy way for applications to store and retrieve default values, or values to save and restore upon request for the parameters they need.

The Setting Repositories are defined by :

  •  Setting attribute values which are defined and stored using setting repositories.
  •  Setting repositories are made up of setting metadata XML file & setting file
  • A Setting Metadata XML File
    •  Setting repositories are first defined in XML files.
    •  For any attribute following values are defined in the XML
      •  attribute name, alias, type, range, step, default value, etc.
  •  A Setting File
    •  Setting Attribute values are persisted in these files.
    • A setting file has a .CATSettings extension.

More about Setting Metadata XML File

There is an XML file for each setting repository which contains the setting repository and attributes metadata.

This XML file is used, for example, whenever a setting attribute is read or written, to check whether a definition for this setting attribute exists, and the consistency of the read or write operation against this definition.

The XML files are located in the run time view : 

  • os_x\resources\SettingsMetadata

 Please refer to the article "Settings Creation Guidelines" [2] where are detailed the steps to create setting repositories and attributes.

General Information about Preferences Page implementation

The difference between old preference page and new page can be summarized as following:

  • the old tabs are now replaced by collapsible sections
  • The New Preferences supports all basic controls.
  • Manage dependency between controls
    • Manage availability/contents/actions of dependent controls
  • Support Admin lock

Concerning the user workflow the page has Ok, Apply, Close & Reset buttons :

  • Ok – Save all modifications & close panel
  • Apply – Save modifications done so far & keep panel open
  • Reset – Cancel all modifications
  • Close -  Close the panel

 

Concerning the declarative files :

  • One declarative file for each tab page (collapsible section)
  • Reuse of information from SettingsMetadata files

 

General Information for Preferences page creation

Let's see how to create a new preference page with a set of collapsible sections, in the "New Reference" panel.

The CAA App developer needs to create a directory for their particular tree node with directory name same as node name in preference organization tree.

 Create the directory structure similar to tree structure inside \resources\CAfrComponents\

When node is selected, only pages present in the directory with name same as selected node will be displayed.

For each collapsible section, an xml file will be created.

Please refer to [3] for detailing sample use case illustrating how to create a new preference page.

Preference Page XML Structure

We now support agnostic format instead of directly using C++ class names as XML tags. 

 CATCAfrPreferencePage -> preferencepage
 CATAfrPrefItemGroup -> preferencegroup
 CATAfrPrefItem -> preferenceitem
 CATAfrMorePreferences -> morepreferences

Some of the old XMLs created might be using class names. Please replace these class names with agnostic tags now on. .

 Below is an example of the xml of a page saved as ‘TestPage.xml' 

<?xml version="1.0"?>
<Styles xmlns:syp="http://www.3ds.com/xsd/SYP">
  <Template syp:name="TestPagesettings" Target="CATAfrComponent">
    <preferencepage Name="General" Resource="TestMePreferences" RepositoryName="GeneralGeneral">
      <preferencegroup Name="Group1"  IconName="I_Icon1">
        <preferenceitem Name="DemoMode"  Lock="1"/>
      <preferencegroup/>
      <preferencegroup Name="Group2"  IconName="I_Icon1">
        <preferenceitem Name="DragAndDrop"/>
        <preferenceitem Name="MaxDisplay" RepositoryName="GeneralFind" Resource="GeneralFind"/>
      <preferencegroup/>
    </preferencepage>
    </Template>
</Styles>

 

The XML file must be saved inside ..\resources\CAfrComponents\..

Please note:  New XML makes use of Attributes defined in SettingsMetadata files

The first statement

The XML starts with following lines

<?xml version="1.0"?>

 Then there is declaration of Styles tag and Template creation.

This is required to process the xml file using style processor.

<?xml version="1.0"?>

<Styles xmlns:syp="http://www.3ds.com/xsd/SYP">

  <Template syp:name="TestPagesettings" Target="CATAfrComponent">

    < !-- All the file content will go here -- >

  </Template>

</Styles>  

 

Important point :

The Template name must always be : xmlfilename + “settings”

E. g. If file is saved as ‘TestPage.xml’ then name of the template must be ‘TestPagesettings’

Do not consider pagePosition as a part of template name.

  • E. g. If page named as “TestPage” should appear at seventh position in ME Preferences list so it should be renamed as “7_TestPage.xml”. But its Template name must always be : “TestPagesettings”

 

Preferencepage Tag

The whole page structure is defined in this tag.

This tag displays it as expander.

<preferencepage

     Name="General"

     IconName="Page_Icon"

     Resource="TestMePreferences"

     RepositoryName="GeneralGeneral">

 ....

 </preferencepage>

Even though it is allowed to add preferencepage tag inside preferencepage, do NOT do it .

 

preferencepage Attributes

Please find below the attributes description associated to a preferencepage :

  • Name: You can give any name. The title corresponding to this name is read from the NLS file. So this name and NLS name must match.

  • IconName [Optional]: This is the name of the icon which appears on the expander

  • Resource : The name of the resource file where all information of this page is defined. E.g. Title.

  • RepositoryName : There must be a XML file with this name in SettingMetadata directory (win_b64\resources\SettingsMetadata\).All informations about controls defined in this page are retrieved from this metada XML file.

     IconName="I_Icon1"

     Align="1"

     Resource="GeneralFind"

 ...

 </preferencegroup>

 

preferencegroup Attributes

Please find below the attributes description associated to a preferencegroup :

  • Name :You can give any name. The title corresponds to this name is read from the NLS file. So this name and NLS name must match. NLS files are read from standard resources directories (resources\msgcatalog).

Note- Page owner have to make sure that each group must have unique name. Otherwise unnamed groups will have same default

name.

  • IconName [Optional]:This is the name of the icon which appears with the group. Icon files are read from standard resources directories (resources\graphic\icons). 
  • Align [Optional ]
    • Default value=0
      •  By default controls will be placed vertically i.e. one below another
    •  If the value of this option is "1" then all controls in this group will be arranged in a flow layout
      • Controls are placed one after another
      •  Depending on window size, they are shown at next line
  •  Border [Optional ]
    • Default value=1  
    •  If the value of this option is 0 then group border is not visible. This option is very useful to arrange the controls horizontally to have grid like effect. Check below example.

  <preferencegroup Name="ContextPriorityGroup"  >

  

   <preferencegroup Align="1" Border="0" >

    <preferenceitem Name="ContextPriority"  />

    <preferenceitem Name="DeepSearch" />

    <preferenceitem Name="DeepSearch" />

   </preferencegroup> 

 

   <preferencegroup Align="1" Border="0" >

    <preferenceitem Name="ContextPriority"  />

    <preferenceitem Name="DeepSearch" />    

   </preferencegroup>

 </preferencegroup>

 

  • RepositoryName [Optional] : You need to use this options when you want to combine the settings from more than one Settings Repository.
    • Value is the name of setting metadata file where attribute node with same name as defined by ‘Name’ is present.
    • If this field is mentioned, it will override the Setting Repository mentioned in preferencepagetag for Items in this group.

 

  •  Resource [Optional] :Value is the name of resource file where all title/label values for this control are defined.
    • You need to use this options when you want to combine the values from more than one Resource file.
    • If this field is mentioned, it will override the Resource file mentioned in preferencepage tag for Items in this group.

 

preferenceitem Tag

Elements defined with this tag are displayed as controls on the page.

  •  You cannot use this tag as a container.
  •  It must be used inside preferencegroup tag or preferencepage tag as per the requirements.

<preferenceitem

     Name="ProfileName"

     RepositoryName="UserProfileRepository"

     Resource="TestMePreferences"

     Lock="1"

     UserDefinedCtr="Combo"

     MoreInfo=""

     />

 

preferenceitem Attributes

Please find below the attribute description associated to a preferenceitem :

  • Name :There must be an attribute node inside the SettingMetaData file defined by ‘RepositoryName’ with same name as this. The title corresponding to this name is read from the NLS file. NLS Title will be displayed as label of this control. If there is no attribute node with this name then message is displayed in page UI. preferenceitem Attributes

  NOTE- At some places in SettingMetaData, it is observed that the 'Name' is more discriptive and contain more than one

  word separated by space. In such case actual attribute used to read and write data in settings is mentioned using

  'alias'. In such cases use alias name in XML.

  Examples-   

  <st:attribute Name="Geometry Quality Enhancement Mode" alias="GeomQualityMode">

  <st:attribute Name="Show/NoShow dialog box" alias="ShowCompletionDialogBox">

  <st:attribute Name="Join surfaces of the model" alias="ApplyJoin">

  In some cases both 'Name' and 'alias' are single word and are not same, In such cases XML creator has to confirm which

  attribute is used to read and write data in settings.

  Examples-   

 <st:attribute Name="Units" alias="UnitsMode">

 <st:attribute Name="Activation" alias="MemWarningActive">

 <st:attribute Name="HighlightFace" alias="ZBuffer">

 

  • Lock [Optional]: By default, ‘Lock=1’ which means admin lock is displayed in front of that control. If user does NOT want to show lock then set ‘Lock=0’.Interpretation of lock is same as before. 

 

  •  RepositoryName [Optional] :You need to use this option when you want to combine the settings from more than one Settings Repository.Value is the name of setting metadata file where attribute node with same name as defined by ‘Name’ is present. If this field is mentioned, it will override the Setting Repository mentioned in preferencepage tag for this Item.

 

  • Resource [Optional] : Value is the name of resource file where all title/label values for this control are defined. You need to use this options when you want to combine the values from more than one Resource file. If this field is mentioned, it will override the Resource file mentioned in preferencepage tag for this Item

 

  •  UserDefinedCtr [Optional] :  More information about this is mentioned on slide titled ‘preferenceitem UI generation’.

 

  • MoreInfo [Optional]: More information about this is mentioned on slide titled ‘More customizations for controls'. Multiple options can be added separated by space. Example - MoreInfo="PlaceHolderControl RuntimeValues"

 

  •  Indent[Optional] : User can indent a component little outwards from margin by mentioning Indent="1".

  • IconName[Optional] :This property is useful to change the icon at run-time. Use it in trigger.

  •  Visibility[Optional] : This property is useful to change the visibility of the control.

  • EnableFlag[Optional] : This property is useful to change the sensitivity of the control.

  • ControlValue[Optional] : This property is useful to Get/Set value of the control. Use it in trigger.

  • ValueChanged[Optional] : This is read only property useful to get value change notification. Use it in trigger.

 

 morepreferences Tag

This tag is introduced to allow user to hide a set of preferences in default view & show only if user wants it

By default a button ("More Options") will be shown. On clicking this button, controls under morepreferences tag will be displayed

On expanding this set, a new button ("Less Options") to hide the set will be displayed.

User can set label of buttons through CATNls files.

It must be used inside preferencegroup tag or preferencepage tag as per the requirements.

It is recommended not to use 'morepreferences' tag inside the group for which Align="1" or Grid="1". This is because this option will be arranged with other controls from the group. And its UI will not be uniform across the pages.

It is recommended not to use 'morepreferences' inside 'morepreferences'. Define it only at level 1.

Defining Titles - For example, In XML entry is

<morepreferences Name="MoreDisplay">

…

</morepreferences >

 

Then define its NLS file entry as below.

MoreDisplay.UncheckState.Title="More Display Options";

MoreDisplay.CheckState.Title="Less...";

MoreDisplay.ShortHelp="More Button ToolTip Test"; //ToolTip

 If NLS titles are not defined for 'morepreferences' tag used in XML then default Titles used are "More Options" and "Less Options".

Customizing File chooser and folder selector

 User can customize file filters and can make editor enabled or disabled for manual edition. By default it will be enabled.

For this user need to add entries similar to below in the CATRsc file. In case of file browser the first filter entry will be the default filter.

XML file entries : 

<preferenceitem Name="FileBrowse"  UserDefinedCtr="FileChooser" />
<preferenceitem Name="FolderSelect"  UserDefinedCtr="FolderChooser" />

CATRsc file entries : 

FolderSelect.Enable="0";
//FileBrowse.Enable="0";
FileBrowse.FileDescriptor0="Text Files";
FileBrowse.FileExtension0="*.txt";
 
FileBrowse.FileDescriptor1="xml Files";
FileBrowse.FileExtension1="*.xml";
 
FileBrowse.FileDescriptor2="Unix Script Files";
FileBrowse.FileExtension2="*.Sh;*.bsh";
 
FileBrowse.FileDescriptor3="C++ Source Files";
FileBrowse.FileExtension3="*.h;*.hpp;*.hxx;*.cpp;*.cxx;*.cc";

Creating NLS Files

As per the existing infrastructure you need to put the NLS file in directory ..\resources\msgcatalog\

Titles/Labels are searched in the file identified by ‘Resource’ property of TAG from XML.

Define the titles in NLS as – ControlName.Title = “Control Title”;

There are some rules for defining values for some specific controls which are explained on next slide

Check additional info at Resource Info

 

Defining values for specific controls in NLS Files

  •  RadioButton

ControlName#N.Title = “Control Title”

Example :

If control name is ContextualPriority and 2 radio buttons are created for it then add entries similar as below in NLS file.

  •  ContextualPriority0.Title = "Technical Documentation";
  • ContextualPriority1.Title = "User Companion";

 

  •  ComboBox

ControlName#N.Title = “Combo item value”

Example :

If  control name is OnlineHelp and there will be 2 items in Combo Box then add entries similar as below in NLS file.

  • OnlineHelp.Title = "Online Help combo "; //Combo Label
  •  OnlineHelp0.Title = "Online Help1 ";
  • OnlineHelp1.Title = "Online Help 2";

Note- If Combo/Radio values are not defined in NLS file then control will not appear in the page. Instead message to

add values in NLS will appear.

Combo/ Radio Value Skipping Support

User can skip values from combo/ radio list by skipping the corresponding entries from CATNls file.

 Example :   

GPColor1.Title="TestA";
GPColor5.Title="TestB";
GPColor8.Title="TestC";
GPColor10.Title="TestD";
GPColor.MaxIndex="10";
GPColor.Default="5";

 

In the above example values for index 0, 2-4, 6-8 and 9 will have no existence.

NOTE: It is mandatory to mention “MaxIndex” for value skipping to work. Default value should be #n and not the string.

In above example user wants to set default value "TestB" so the default value index is defined as GPColor.Default="5";

In case of [Int+enum], in some cases setting metadata entries starts from 1 to #N and not from 0. In such cases define NLS entries starting with first entry ControlName1.Title="First entry label". i.e. ControlName#n.Title="Nth entry label";

In existing preferences there are controls which are added to the page but there value is not saved in repository.

For these control there is no entry in the setting metadata file, that's why  it is not possible to create such type of controls with new infra.

To overcome this we have provided option MoreInfo="PlaceHolderControl" which can be used along with different 'UserDefinedCtr' options.

<preferenceitem Name="inPlaceColorCombo"   MoreInfo="PlaceHolderControl" UserDefinedCtr="ColorCombo"   />

<preferenceitem Name="inPlaceBool"   MoreInfo="PlaceHolderControl" UserDefinedCtr="Bool"   />

<preferenceitem Name="inPlaceTextBox"   MoreInfo="PlaceHolderControl" UserDefinedCtr="TextEdt"   />

Dependent Control Handling

Consider group in following picture.

 

There are 10 different values in combo "Default Prefix...", Prefix0, Prefix1, Prefix2,.. Prefix9.

We want to manage following dependency based on Check/Uncheck of "ContextPriority" checkbox:

  • "ContextPriority" Uncheck:
    • "Default Prefix.." combo should show Prefix0
    • "Deep Search Activation" checkbox should be Unchecked
    •  "Maximum Display Result" slider should be disabled with value = 100
    •  "Maximum Pre Highlight..." slider should be enabled with value = 10

 

  •  "ContextPriority" Check:
    •  "Default Prefix.." combo should show Prefix2
    • "Deep Search Activation" checkbox should be Checked
    • "Maximum Display Result" slider should be enabled with value = 3000
    • "Maximum Pre Highlight..." slider should be enabled with value = 4900

This can be achieved using following syntax in the XML file.

<?xml version="1.0"?>

<Styles xmlns:syp="http://www.3ds.com/xsd/SYP">

 <Template syp:name="GeneralFindsettings" Target="CATAfrComponent">

  <preferencepage Name="GeneralFind" Resource="TestMePreferences" RepositoryName="GeneralFind">

   <preferencegroup Name="PrefixGroup" >

    <preferenceitem syp:name="Prefix" Name="Prefix"   />

   </preferencegroup>

   <preferencegroup Name="ContextPriorityGroup" >

    <preferenceitem syp:name="ContextPriority" Name="ContextPriority"  />

   </preferencegroup>

   <preferencegroup Name="DeepSearchGroup"  >

    <preferenceitem syp:name="DeepSearch" Name="DeepSearch"  />

   </preferencegroup>     

   <preferencegroup Name="MaxDisplayGroup" Align="1" >

    <preferenceitem syp:name="MaxDisplay" Name="MaxDisplay"  />

    <preferenceitem Name="MaxPSO"  />

   </preferencegroup>

   <preferencegroup Name="MaxPSOGroup" >

    <preferenceitem syp:name="MaxPSO" Name="MaxPSO"  />

   </preferencegroup>  

  </preferencepage>

  <CATSYPTriggerCollection>

   <CATSYPTrigger Property="ControlValue" TargetName="ContextPriority" Value="0">

    <CATSYPSetter Property="ControlValue" TargetName="DeepSearch" Value="0" />

    <CATSYPSetter Property="EnableFlag" TargetName="MaxDisplay" Value="0" />

    <CATSYPSetter Property="ControlValue" TargetName="MaxDisplay" Value="100" />

    <CATSYPSetter Property="ControlValue" TargetName="Prefix" Value="1" />

    <CATSYPSetter Property="ControlValue" TargetName="MaxPSO" Value="10" />

   </CATSYPTrigger>

   <CATSYPTrigger Property="ControlValue" TargetName="ContextPriority" Value="1">

    <CATSYPSetter Property="Visibility" TargetName="DeepSearch" Value="1" />

    <CATSYPSetter Property="ControlValue" TargetName="DeepSearch" Value="1" />

    <CATSYPSetter Property="EnableFlag" TargetName="MaxDisplay" Value="1" />

    <CATSYPSetter Property="ControlValue" TargetName="MaxDisplay" Value="3000" />

    <CATSYPSetter Property="ControlValue" TargetName="Prefix" Value="3" />

    <CATSYPSetter Property="ControlValue" TargetName="MaxPSO" Value="4900" />

   </CATSYPTrigger>

  </CATSYPTriggerCollection>

 </Template>

</Styles>

Tags used for "Dependent Control Handling" mechanism:

  • syp:name=".." need to be set on all preferenceitem for which user need to manage dependency

  • CATSYPTriggerCollection : This tag is used to define set of dependencies to be managed

  • CATSYPTrigger : This tag is used by preferenceitem on which other preferenceitems are dependent. It's properties are,

    • Property : This defines whether user want to control the value ("ControlValue") or enable/disable state ("EnableFlag")

    • Value : Value to set for control depending on the Property type

    • TargetName : Value of "syp:name" property of preferenceitem

  • CATSYPSetter : This tag is used by preferenceitems which are are dependent on other preferenceitem. It's properties are similar to CATSYPTrigger

 

Important points

For radio buttons, the button count starts from 0.

Example:

If there are three radio buttons and first button 3rd button is selected.

Now depending on the other control value, user wants to set first button as selected then in xml file set control value to 0. Value="0".

For combo, Index starts from 0.

  • In case of combo, user needs to specify index value in the XML.

For color picker

  • User must define all 4 values aka RGBA.

<CATSYPTriggerCollection>

  <CATSYPTrigger Property="ControlValue" TargetName="BGMode" Value="0">

   <CATSYPSetter Property="ControlValue" TargetName="Background" Value="255 0 0 255" />

   <CATSYPSetter Property="ControlValue" TargetName="NoShowBackgroundRGB" Value="0 255 0 255" />

  </CATSYPTrigger>

  <CATSYPTrigger Property="ControlValue" TargetName="BGMode" Value="1">

   <CATSYPSetter Property="ControlValue" TargetName="Background" Value="255 255 255 255" />

   <CATSYPSetter Property="ControlValue" TargetName="NoShowBackgroundRGB" Value="0 0 0 255" />

  </CATSYPTrigger>

</CATSYPTriggerCollection>

Dependent value is calculated even with fresh launch.

Triggers are fired from top to bottom. So If there are more than one trigger which change the value of control then the value of control can vary depending on the order of the triggers defined.

Do not define the cyclic dependency otherwise behavior is unpredictable.

 

Common Settings

Define a preference once and use them at several places.

There is a provision to create a XML file which can be included in other XMLs defined for Preference pages

Location where to be included XML files should be kept

  • It is mandatory to put pages in directory ..\resources\CAfrComponents\IncludePreferences

The target of include page template must be ‘CATCAfrInclude’

The Preference page xml that wants to use common setting must include the file in which template of interest is defined. Including Settings from Other Pages.

Example:

  1. Create an XML file which will be included in preferences XML file (e.g. IncludePage.xml)

  2. Put the file ‘IncludePage.xml’ in directory  ..\resources\CAfrComponents\IncludePreferences 

  3. Include it in preferences XML file (e.g. GeneralIncludeTest.xml)

  4. Put the file ‘GeneralIncludeTest.xml’ in …\resources\CAfrComponents\ImportantPreferences\Topic1\GeneralIncludeTest.xml

  5. Compile and launch application to test page

  6. Same include XML file (IncludePage.xml) can be included in any other preferences XML file as per need

Inter Tab dependency

If user want to modify the value of control on page2 with change in the value of the control on page1 then user need to add the same control on page1 similar to the control on page2. But mark it as invisible.

Example: User wants to change the value of BGMode from page2 with change in value of BorderEdgesMode on page1. For this user need to create page XML similar to below.

Content of Intertab_TestPage1.xml

<?xml version="1.0"?>
<Styles xmlns:syp="http://www.3ds.com/xsd/SYP">
 <Template syp:name="Intertab_TestPage1settings" Target="CATAfrComponent">
   <preferencepage Name="Page1" Resource="TestResources" RepositoryName="ABC">
     <preferenceitem syp:name="BorderEdgesMode" Name="BorderEdgesMode" />
     <preferenceitem syp:name="BGMode" Name="BGMode" Visibility="0" RepositoryName="PQR" Resource="TestResources2"/>
   </preferencepage>
   <CATSYPTriggerCollection>   
    <CATSYPTrigger Property="ControlValue" TargetName="BorderEdgesMode" Value="0">
      <CATSYPSetter Property="ControlValue" TargetName="BGMode" Value="0" />
    </CATSYPTrigger>
    <CATSYPTrigger Property="ControlValue" TargetName="BorderEdgesMode" Value="1">
      <CATSYPSetter Property="ControlValue" TargetName="BGMode" Value="1" />
    </CATSYPTrigger>   
  </CATSYPTriggerCollection>
 </Template>
</Styles>

Here with change in value of BorderEdgesMode, the value of BGMode is changed and saved to in memory copy of the PQR Repository.

The change will be automatically reflected on Page2 after user collapse and expand (Refresh) Page2. Here Page2 reads the updated value from the PQR Repository.

content of Intertab_TestPage2.xml

<?xml version="1.0"?>
<Styles xmlns:syp="http://www.3ds.com/xsd/SYP">
 <Template syp:name="Intertab_TestPage2settings" Target="CATAfrComponent">
   <preferencepage Name="Page1" Resource="TestResources2" RepositoryName="PQR">
     <preferenceitem syp:name="BGMode" Name="BGMode" />
   </preferencepage>
 </Template>
</Styles>

Adding Callback/Notifications

We have provide option to add callbacks on OK, Apply, Cancel/Close and Reset buttons. With this added callback method will be called when that particular button is pressed. User can add post processing in this callback method.

IMPORTANT NOTE- Below callback will be dispatched on OK, APPLY, CLOSE OR RESET button click. And this will be sent
     to all subscribers of particular event Even though the user concerned repository is not modified.
  
     So for more precise control we recommend to use CATSettingRepository events such as 
     Repository_Modified or CATSysSettingRepository_Updated

User can add callbacks by adding below lines in there code.

OK Button: To add callback on OK button press event, add below lines.    
::AddCallback(this,
               CATCAfrMePreferencesNotificationManager::GetPreferencesNotificationManager(),
                   CATCAfrMePreferencesNotificationManager::SETTINGS_SAVED(),
                   (CATSubscriberMethod)&MyClass::MyCBMethod,
                   NULL);

Apply Button: To add callback on Apply button press event, add below lines.

  ::AddCallback(this,
               CATCAfrMePreferencesNotificationManager::GetPreferencesNotificationManager(),
                   CATCAfrMePreferencesNotificationManager::SETTINGS_APPLIED(),
                   (CATSubscriberMethod)&MyClass::MyCBMethod,
                   NULL);

Close Button or Window Close: To add callback on Close button press event, add below lines.

::AddCallback(this,
               CATCAfrMePreferencesNotificationManager::GetPreferencesNotificationManager(),
                   CATCAfrMePreferencesNotificationManager::SETTINGS_CANCLED(),
                   (CATSubscriberMethod)&MyClass::MyCBMethod,
                   NULL);

Reset Button: To add callback on Reset button press event, add below lines. (This button yet not available).

  ::AddCallback(this,
               CATCAfrMePreferencesNotificationManager::GetPreferencesNotificationManager(),
                   CATCAfrMePreferencesNotificationManager::SETTINGS_RESET(),
                   (CATSubscriberMethod)&MyClass::MyCBMethod,
                   NULL);

 

Combo Image Combo

User can define Image Combo as follows :

  • Index for defining ImagePath of images starts with 0 and cannot be skipped.

  • Corresponding ImageStoreValue should be mentioned if user want to store values other than Image index.

  • If ImageStoreValue is defined then user can mention corresponding storage value for an image in Default to make it selected by default.

  • If ImageStoreValue is not defined then user can mention corresponding index value for an image in Default to make it selected by default.

  • Default ImageHeight and ImageWidth is set to 25 X 25. It cannot be more than 100 X 100.

 

Example:

XML entry:

<preferenceitem syp:name="ImageCombo" Name="ImageCombo"  Lock="1" UserDefinedCtr="ImageCombo"/>

CATRsc entry:

ImageCombo0.ImagePath="Preferences\\Pref_homecompass_1.png";   //0 is image index

ImageCombo1.ImagePath="Preferences\\biolma_ap_appicon22.PNG";

ImageCombo2.ImagePath="Preferences\\catlma_ap_appicon22.PNG";

ImageCombo3.ImagePath="Preferences\\dellma_ap_appicon22.PNG";

ImageCombo4.ImagePath="Preferences\\I_Display_On.PNG";

 

//5 is storage value against image at index 0. If storage value is not mentioned then image index is used as storage value.

ImageCombo0.ImageStoreValue="5";

ImageCombo1.ImageStoreValue="10";

ImageCombo2.ImageStoreValue="15";

ImageCombo3.ImageStoreValue="20";

ImageCombo4.ImageStoreValue="25";

 

//Default selected image

ImageCombo.Default="20";

//Image Dimensions. Max supported Dimensions is 100 X 100.

ImageCombo.ImageHeight="50";

ImageCombo.ImageWidth="100";

Conditional variable support

User can manage the availability/Visibility of the control OR group control by creating separate template and including it on specific condition. Only few functions are supported by this syntax and are listed below.

Fuctions:

  • IsProductAuthorized - indicates whether a given product is available depending on the current license

  • IsMCSProductAuthorized - indicates whether a given MCS product is available depending on the current license

  • IsFrameworkAuthorized - indicates whether a given framework is packaged or not depending on the current license

  • IsOptionAuthorized - indicates whether a given licensing option is activated or not

  • GetEnv - retrieves the value of the given environment variable. If the variable is not defined, GetEnv returns 0.


Operator: The following operators are supported by CAfr:

== (equal operator)

!= (not-equal operator)

&& (logical And operator: &&. In XML file only the syntax && is supported. && is not supported.')

|| (logical Or operator)

! (negation operator)

 

Between each of these elements you can put an optional blank space. If the expression you put in the Condition attribute is ill-formed an assertion will occur when the condition expression will be processed.


The following list of condition expression are some valid samples:

         $var == some_value

         some_value!=$var

         (some_value==$var) &amp;&amp; $var2

         ($var==some_value) &amp;&amp; !IsProductAuthorized(my_product)

         $var|| $var2

 

In Short

The New reference panel revisit the command Preference, and allow user to customize settings panel without need to develop any code, just by filling an xml file dedicated for this.


Thanks to this new infrastructure, user will have an more easy access to the most used and most important preferences, and will keep an organised access to his apps prefences.
For creating a new Preference Page in the "New Preferences" command dialog box, user will need to create a directory structure similar to tree structure inside \resources\CAfrComponents.

The preferences Page contains a set of Preference group (collapsible section) and each section regroups a set of Preference item dedicated to model parameters with their values.

The methodology enabling user to customize the "New Preferences" dialog panel, and review of all XML tags and attribute value is fully detailed in this article.
